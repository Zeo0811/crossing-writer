# v1.5.0 UI Redesign — Design Spec

- **Spec ID**: v1.5.0-ui
- **Date**: 2026-04-15
- **Status**: Brainstorm-approved, awaiting per-checkpoint mockup approvals
- **Predecessor**: SP-14 (visual tokens approved)
- **Successor**: 2.0.0 (final polish)

---

## 1. 动机

1.0.0 跌跌撞撞跑通端到端真实测试，暴露的疼痛点全在 UX：
- 阶段切换"全靠刷新"，缺"下一步"CTA
- error / blocked 等态视觉缺失，"啥也看不到"
- agent 执行过程不透明，卡住不知原因
- 编辑动作（rewrite/@-mention/图片）手感差
- 项目列表无归档/筛选，配置区入口分散

1.5.0 不动后端契约，**只重做 UI 与交互流程**，下一步才进 2.0.0 的细节打磨。

## 2. 锁定决策

| 维度 | 选择 |
|---|---|
| 视觉语言 | 沿用 SP-14 v5 retro-pixel（dark 默认 + light 可选） |
| 走查顺序 | 按用户动线 |
| Mockup 形态 | 完整可交互 prototype（路由切屏 + 状态切换器 + mock 数据） |
| 承载位置 | `packages/web-ui/` 内，`?mock=1` URL 开关进 mock 路由层，**不污染真实 API 路由** |
| 剧本 | 5 个示例项目 + 1 个虚构主角「测评 Cursor IDE」可切全 state |
| 优先级靶子 | a 流程清晰度 → b 状态全态 → c agent 透明度 → d 编辑手感 → e 列表/归档 → f 配置区 |

## 3. Mock 架构

- **入口**：访问 `?mock=1` 后整站切到 `MockShell`，所有数据来自 `src/mock/*.ts`
- **状态切换器**：右下浮动 `MockStateSwitcher`，可切 project status、agent 阶段、错误注入
- **mock 数据集**：
  - `mockProjects[]`：5 个不同状态的示例项目
  - `mockHero`: 虚构主角项目「测评 Cursor IDE」，承载所有 state 演示
  - `mockEvents[]`：SSE 事件流回放
  - `mockKnowledge[]`、`mockStylePanels[]`、`mockAgents[]`
- **路由**：复用 React Router；mock 模式下 wrap 一层 `<MockProvider>` 注入 fixture
- **隔离**：mock 组件命名 `Mock*`，迁移完成后由对应正式组件替代；mock 文件统一放 `src/mock/`，方便 1.5.0 收尾时整体删除/保留

**节奏**：每 checkpoint 一回合 brainstorm → mockup 实现 → 浏览器验收 → 迭代 → 下一个。

## 5. 验收标准

每个 checkpoint mockup 验收：
- 所有 state 变体可点击切换
- 所有"下一步"CTA 明确可达
- error / loading / empty 全态可见
- 与 SP-14 token 一致（无硬编码 hex）
- 在 dark 与 light 两套主题下都正常渲染

10 个 checkpoint 全数批准 → 拆 plan → subagent-driven 把 mock 替换为真实 API → 1.5.0 release。

## 6. 不在本 spec 范围

- 后端 API 变更（如有发现耦合问题，单独开 ticket）
- 多语言（保持中文）
- 移动端适配（桌面优先，min-width 1280）
- 微信发布的真实凭证流（mock 用假 token）
